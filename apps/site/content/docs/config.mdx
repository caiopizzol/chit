Chit has one public config surface: `chit.config.json`.

The schema answers two questions:

1. Which local adapters and models can routines use?
2. Which routines can this project run?

<Callout title="The split">
	`profiles` bind names to local adapters and models. `routines` declare the workflow.
	A routine agent points to a profile, so the workflow stays separate from the local model choice.
</Callout>

## Schema At A Glance

This is the shape to understand first. `profiles` declare local model bindings. `routines` declare what runs.

## Profiles

Profiles are machine-local. They let the same routine run with different adapters or models without editing the routine.

| Shape | Example | Meaning |
|---|---|---|
| adapter | `"critic": "gemini"` | Use the adapter default model. |
| adapter:model | `"builder": "codex:gpt-5.5"` | Use one explicit model. |
| object | `"judge": { "adapter": "claude", "model": "haiku" }` | Same binding, written explicitly. |

Built-in adapters are `claude`, `codex`, and `gemini`. Custom adapters must use object form.

```json
{
	"profiles": {
		"planner": "claude:sonnet",
		"builder": "codex:gpt-5.5",
		"critic": "gemini",
		"local": { "adapter": "my-adapter", "model": "custom-model" }
	}
}
```

## Routines

A routine is a declared workflow. It can be text-only, composed from other routines, looped, check-only, sandboxed, or human-gated. Chit derives that from the fields.

<TypeTable
	type={{
		input: { type: "string", description: "Shorthand for one required string input." },
		inputs: { type: "array | object", description: "Multiple inputs, or full input definitions." },
		agents: { type: "object", description: "Participants inside the routine. Required when the routine has call steps." },
		steps: { type: "array", required: true, description: "Ordered steps." },
		repeat: { type: "object", description: "Loop stop condition and cap." },
		output: { type: "string", description: "Step id used as the routine output." },
		limits: { type: "object", description: "Per-call and whole-run time bounds." },
	}}
/>

<Callout title="Names are yours">
	`builder`, `critic`, `planner`, `judge`, and `griller` are not built into Chit.
	They are ids you choose.
</Callout>

## Steps

Every step has an `id`. One other field chooses the step kind.

| Step kind | Required fields | Output | Description |
|---|---|---:|---|
| `call` | `id`, `call`, `prompt` | yes | Calls a routine agent. |
| `ask` | `id`, `ask` | yes | Pauses for one operator answer. |
| `check` | `id`, `check` | yes | Runs one or more commands. |
| `format` | `id`, `format` | yes | Builds text from templates. |
| `routine` | `id`, `routine`, `inputs` | yes | Runs another routine. |

```json
[
	{ "id": "plan", "call": "planner", "prompt": "Plan:\n{{ inputs.idea }}" },
	{ "id": "approve", "ask": "Adjust this plan?\n{{ steps.plan.output }}" },
	{ "id": "build", "routine": "implement", "inputs": { "task": "{{ steps.plan.output }}\n{{ steps.approve.output }}" } },
	{ "id": "verify", "check": "bun test" }
]
```

Checks can be strings or argv objects.

```json
{ "id": "verify", "check": "bun test" }
```

```json
{ "id": "verify", "check": [{ "command": "bun", "args": ["test"] }] }
```

## Templates

Prompts, formats, ask questions, and routine inputs can read prior context.

| Template | Meaning |
|---|---|
| `{{ inputs.task }}` | A user input passed to the routine. |
| `{{ steps.plan.output }}` | Output from an earlier step. |
| `{{ steps.verify.output }}` | Check result from an earlier step. |
| `{{ iteration }}` | Current loop iteration. |
| `{{ diff }}` | Current sandbox diff. |

## Loops

Loops live under `repeat`. The stop condition is `repeat.until`.

| Condition | Example | Best use |
|---|---|---|
| checks pass | `"checks-pass"` | Implementation and verification loops. |
| exact step output | `{ "step": "review", "equals": "pass" }` | A model or human verdict controls completion. |
| all conditions | `{ "all": ["checks-pass", { "step": "review", "equals": "pass" }] }` | Checks and review both block convergence. |

```json
{
	"repeat": {
		"until": { "all": ["checks-pass", { "step": "review", "equals": "pass" }] },
		"maxIterations": 3
	}
}
```

A judged loop (`{ step, equals }`, or an `all` that contains one) must set `maxIterations`. A `checks-pass` loop may omit it; Chit uses a default cap of 5 iterations.

<Callout title="Prefer checks when they can prove the result">
	Model verdicts are useful, but deterministic checks are the strongest convergence signal.
	Use a blocking model verdict only when judgment is the point of the routine.
</Callout>

## Validation

Two layers check config.

| Layer | What it does |
|---|---|
| JSON Schema | Editor hints and shape validation. |
| Parser | Authoritative runtime validation before any model is called. |

The schema's profile rules are generated from the same adapter registry the parser uses, so built-in adapter/model rules stay aligned.

The parser rejects, among others:

| Rejected | Example | Reason |
|---|---|---|
| Unknown top-level field | `{ "recipes": {} }` | Allowed: `$schema`, `profiles` (or `agents`), `routines`. |
| `profiles` and `agents` together | both keys set | They are aliases. Use one. |
| Non-kebab routine id | `"Build_It"` | Ids match `^[a-z][a-z0-9-]*$`. |
| Unknown adapter shorthand | `"x": "ollama:llama3"` | Custom adapters must use object form. |
| Impossible built-in model | `"x": "codex:sonnet"` | `sonnet` is a Claude model, not a Codex model. |
| Trailing colon | `"x": "codex:"` | Use `"codex"` for the default model. |
| Unknown agent field | `{ "adapter": "claude", "temperature": 1 }` | Only `adapter` and `model`. |
| Escaping routine path | `{ "file": "../secret.json" }` | Routine files cannot escape the project. |
| Non-positive `maxIterations` | `0` | Must be a positive integer. |
| A routine with no `steps` | `{ "input": "task" }` | `steps` must be an array. |
| Judged loop with no cap | `{ "until": { "step": "review", "equals": "pass" } }` | Judged loops must set `maxIterations`. |

<Callout title="Structural, not live">
	Validation proves the config is well-formed. It does not prove your local account can use a model.
	`codex:gpt-5.5` is structurally valid, but still fails at execution if the local Codex CLI cannot use that model.
</Callout>

## Compatibility

New configs should use `profiles`, routine `agents`, and agent `profile`.

Chit still accepts older aliases where needed, but the preferred shape is the one documented here.
