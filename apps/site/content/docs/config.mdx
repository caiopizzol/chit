Chit has one public config surface: `chit.config.json`.

<Callout title="New to Chit?">
	Start with the [Get started](/docs) guide for the install-to-first-run path. This page is the field reference.
</Callout>

The config answers two questions:

1. Which local models can routines use?
2. Which routines can this project run?

<Callout title="The split">
	`profiles` bind names to local adapters and models. `routines` declare workflows.
	A routine agent points at a profile, so model choice stays outside the workflow.
</Callout>

## Shape

```json
{
	"$schema": "https://chit.run/schemas/chit.schema.json",
	"profiles": {
		"claude": "claude",
		"codex": "codex"
	},
	"routines": {
		"implement": {
			"input": "task",
			"agents": {
				"builder": { "profile": "claude", "instructions": "Implement the smallest correct change.", "filesystem": "read-write" },
				"reviewer": { "profile": "codex", "instructions": "Review the diff. Return JSON only.", "filesystem": "read-only" }
			},
			"steps": [
				{ "id": "build", "call": "builder", "prompt": "{{ inputs.task }}" },
				{
					"id": "review",
					"call": "reviewer",
					"prompt": "{{ diff }}",
					"json": {
						"schema": {
							"type": "object",
							"required": ["passed", "issues"],
							"properties": {
								"passed": { "type": "boolean" },
								"issues": { "type": "array", "items": { "type": "string" } }
							}
						}
					}
				},
				{ "id": "verify", "check": "bun test" }
			],
			"repeat": {
				"until": { "all": ["checks-pass", { "step": "review", "path": "passed", "equals": true }] },
				"maxIterations": 3
			}
		}
	}
}
```

This is what `chit init implement --template loop` generates. The starter ships the check as a placeholder; swap it for your real command, such as `bun test`.

## Root Fields

| Field | Type | Required | Description |
|---|---|---:|---|
| `$schema` | string | no | JSON Schema URL for editor hints. |
| `profiles` | object | when routines call agents | Local adapter and model bindings. |
| `routines` | object | yes | Named routine definitions. |

## Profiles

Profiles are local. They let one routine run with different adapters or models without editing the routine.

| Shape | Example | Meaning |
|---|---|---|
| adapter | `"reviewer": "codex"` | Use the adapter default model. |
| adapter:model | `"reviewer": "codex:gpt-5.5"` | Use one explicit model. |
| object | `"builder": { "adapter": "claude", "model": "claude-opus-4-8", "effort": "max" }` | Explicit binding with adapter options. |

Built-in adapters are `claude`, `codex`, and `gemini`. Custom adapters must use object form.

| Option | Adapter | Values |
|---|---|---|
| `effort` | `claude` | `low`, `medium`, `high`, `max` |
| `effort` | `codex` | `minimal`, `low`, `medium`, `high`, `xhigh` |

<Callout title="Structural validation">
	Chit rejects impossible built-in pairs like `codex:sonnet` before execution.
	It does not prove your account can use a model. Use `chit doctor --real` for live probes.
</Callout>

## Routines

A routine is an ordered workflow. Chit derives whether it is a text run, flow, loop, or sandboxed run from the fields.

| Field | Type | Required | Description |
|---|---|---:|---|
| `input` | string | no | Shorthand for one required string input. |
| `inputs` | array or object | no | Multiple inputs, or full input definitions. |
| `agents` | object | with `call` steps | Agents local to this routine. |
| `steps` | array | yes | Ordered steps. |
| `repeat` | object | no | Loop stop condition and cap. |
| `output` | string | no | Step id used as the routine output. |
| `limits` | object | no | Per-call and whole-run time bounds. |
| `file` | string | no | In `routines.<id>`, points to a routine JSON file instead of declaring it there. |

Names like `builder`, `reviewer`, `planner`, and `judge` are not built in. They are ids you choose.

## Agents

Routine agents are local to one routine.

| Field | Type | Required | Description |
|---|---|---:|---|
| `profile` | string | yes | Profile id from root `profiles`. |
| `instructions` | string | yes | Stable instructions for this agent in this routine. |
| `filesystem` | string | yes | `none`, `read-only`, or `read-write`. |

## Steps

Every step has an `id`. Exactly one other field chooses the step kind.

| Step kind | Required fields | Description |
|---|---|---|
| `call` | `id`, `call`, `prompt` | Calls a routine agent. |
| `ask` | `id`, `ask` | Pauses for one operator answer. |
| `check` | `id`, `check` | Runs one or more commands. |
| `format` | `id`, `format` | Builds text from templates. |
| `routine` | `id`, `routine`, `inputs` | Runs another routine. |

```json
[
	{ "id": "plan", "call": "planner", "prompt": "Plan:\n{{ inputs.idea }}" },
	{ "id": "approve", "ask": "Adjust this plan?\n{{ steps.plan.output }}" },
	{ "id": "build", "routine": "implement", "inputs": { "task": "{{ steps.plan.output }}\n{{ steps.approve.output }}" } }
]
```

Checks can be command strings or argv objects.

```json
{ "id": "verify", "check": "bun test" }
```

```json
{ "id": "verify", "check": [{ "command": "bun", "args": ["test"] }] }
```

## Structured Call Output

A `call` step can require JSON. Chit parses the model output, validates it against the schema subset, stores the normalized JSON, and can use a field as a loop condition.

```json
{
	"id": "review",
	"call": "reviewer",
	"prompt": "Review the diff. Return JSON only: { \"passed\": boolean, \"issues\": string[] }.",
	"json": {
		"schema": {
			"type": "object",
			"required": ["passed", "issues"],
			"properties": {
				"passed": { "type": "boolean" },
				"issues": { "type": "array", "items": { "type": "string" } }
			}
		}
	}
}
```

The supported schema subset is intentionally small: `object`, `array`, `string`, `number`, `integer`, `boolean`, `required`, `properties`, `items`, `enum`, and `additionalProperties`.

## Templates

Prompts, formats, ask questions, and routine inputs can read prior context.

| Template | Meaning |
|---|---|
| `{{ inputs.task }}` | A user input passed to the routine. |
| `{{ steps.plan.output }}` | Output from an earlier step. |
| `{{ iteration }}` | Current loop iteration. |
| `{{ diff }}` | Current sandbox diff. |

An `ask` step does not store the answer on its own receipt. If you forward the answer into a later routine input or output, it is stored there in plaintext like any other input.

## Loops

Loops live under `repeat`. The stop condition is `repeat.until`.

| Condition | Example | Best use |
|---|---|---|
| checks pass | `"checks-pass"` | Implementation loops with deterministic checks. |
| exact step output | `{ "step": "verdict", "equals": "pass" }` | A constrained text verdict controls completion. |
| structured field | `{ "step": "review", "path": "passed", "equals": true }` | A model returns JSON and one field controls completion. |
| all conditions | `{ "all": ["checks-pass", { "step": "review", "path": "passed", "equals": true }] }` | Checks and review both block convergence. |

```json
{
	"repeat": {
		"until": {
			"all": ["checks-pass", { "step": "review", "path": "passed", "equals": true }]
		},
		"maxIterations": 3
	}
}
```

Judged loops must set `maxIterations`. A `checks-pass` loop may omit it; Chit uses a default cap of 5 iterations.

<Callout title="Checks are the strongest gate">
	Model verdicts are useful when judgment is the point. Deterministic checks are still the strongest convergence signal, but they only prove what they actually exercise.
</Callout>

## Limits

| Field | Type | Default | Description |
|---|---|---:|---|
| `callTimeoutMinutes` | number or `"none"` | 30 | Hard timeout for each model call and check command. |
| `runTimeoutMinutes` | number or `"none"` | 120 | Cooperative wall-time budget for the whole run, checked between steps. |
