title: MCP surface
description: Run a chit manifest one step at a time inside a chat, with a heartbeat on each long step

The MCP surface exposes chit inside a chat through two tool families. Stepwise manifest tools run a manifest one step at a time, with a live heartbeat on each long step. Converge tools drive the autonomous implement/review loop one iteration at a time. Each is the conversational counterpart to a CLI command that otherwise runs to completion in one shot (`chit run` and `chit converge`), so a model and the human watching it can drive and inspect it from the chat.

Install the CLI, then register the MCP server with `chit mcp`:

```sh
bun install -g @chit-run/cli@latest
claude mcp add chit --scope local -- chit mcp
```

`chit mcp` runs the stdio MCP server from the installed `chit` binary (the same binary as `chit run`, `chit converge`, `chit audit`). Upgrade with the same command, `bun install -g @chit-run/cli@latest` (during 0.x, `bun update -g` will not cross a minor). From a source checkout the equivalent is `claude mcp add chit --scope local -- bun <repo>/apps/cli/src/cli/run.ts mcp`.

Source: `apps/cli/src/surfaces/mcp/` (`server.ts` is the tools, `engine.ts` is the stepwise run engine, `converge-engine.ts` is the single-iteration converge driver).

## Execution model

A run is a stepwise projection of a manifest DAG. chit, not the model, decides what is legal to run next.

- **chit owns the order.** A step is ready only when every step in its `manifest.dependencies` is `done`. `chit_run_step` rejects anything else ("not ready; waiting on: X"). The model drives, but it cannot invent routing. There are no dynamic, model-decided handoffs; the static-DAG thesis holds.
- **A step runs exactly once.** `chit_run_step` marks the record `running` synchronously before the first await, so a concurrent call on the same step is rejected ("already running"). On settle the record is terminal: `done | failed | cancelled`. Re-running a terminal step is refused.
- **Completion is all-steps-done**, not just the output step. An independent pending or failed branch keeps the run incomplete.
- **`chit_run_step` blocks** until its step settles. The heartbeat renders on that in-flight tool call, so a model turn is pinned for the step's whole duration.
- **Sessions.** `per_scope` participants need a `scope`. `chit_run_start` rejects a `per_scope` manifest started without one rather than silently running stateless. Within a run, a `per_scope` participant resumes its own session across steps, so a later step can reference its own earlier output without it being re-passed.

## Tools

All results are JSON in a single text block. `describeRun` (embedded in most results) is `{ run_id, manifest, complete, ready[], output? }`, where each `ready` entry is `{ step, kind, participant?, agent?, session? }` and `output` is the final output (present only when `complete`).

### chit_run_start

Start a run. Inputs: `manifest_path` (absolute, or relative to cwd), `inputs` (string to string map, default `{}`), `scope?`, `cwd?` (passed to agents; defaults to the server cwd), `allow_unenforced_permissions` (default `true`). Returns `describeRun` with the initial ready set. Errors: unreadable or invalid manifest, unknown agent, enforcement gap without the flag, `per_scope` without `scope`.

### chit_run_next

List the steps ready to run next (or report completion). Input: `run_id`. Returns `describeRun`.

### chit_run_step

Run one ready step, blocking until it settles; emits a heartbeat while it runs. Inputs: `run_id`, `step_id`. On success returns `{ ran, durationMs, step_output, ...describeRun }`. On cancellation returns `{ cancelled: true, step, durationMs, ...describeRun }` (a clean result, not an error). Rejects out-of-order, duplicate-running, or terminal steps.

### chit_run_cancel

Cancel a step that is currently running. Inputs: `run_id`, `step_id`. Aborts the step's registered controller, the adapter kills the agent's child process, and the step settles `cancelled` (terminal, blocks dependents). Returns `{ step, cancelled, reason?, ...describeRun }` where `cancelled` is `true` if it stopped a running step, else `reason` is `already_done` or `not_running`.

### chit_run_trace

The transcript so far. Input: `run_id`. Returns `{ run_id, manifest, complete, trace[] }`, each trace entry `{ step, kind, participant, agent, status, durationMs, output, error }`.

## Converge tools

The converge tools drive `chit converge` (a write-capable implementer slices the task, a read-only reviewer checks the diff) one iteration at a time, so each iteration is a separate, cancellable tool call and the loop is inspectable between calls. They sit on the same single-iteration primitive and write the same loop log (under chit's state dir, keyed by repo) as the CLI, so a loop driven over MCP is identical on disk to one driven by `chit converge`. The loop log is the durable state; the server keeps an idle-evicting in-memory session for the loop's run-time pieces, never a second source of truth for iterations.

`describeConverge` (the control-plane view, embedded as `loop` in `chit_converge_next` and `chit_converge_cancel` results) is `{ loop_id, scope, cwd, task, max_iterations, iteration, status, active, cancellable, last_verdict?, last_decision?, failure?, audit_refs[], next_action }`, where `status` is `open | running | converged | blocked | max-iterations | cancelled` and `iteration` counts completed iterations.

### chit_converge_start

Open a loop. Inputs: `task`, `scope`, `cwd?` (defaults to the server cwd; also where the loop log is written), `manifest_path?` (default: the built-in converge manifest), `max_iterations` (default 3), `loop_id?`, `force` (overwrite an existing log), `allow_unenforced_permissions` (default `false`). Audit is always on, so the loop records link to the audit transcript. Returns `describeConverge` (plus `warnings[]` for unenforced permissions). Errors mirror `chit converge`: unreadable or invalid manifest, non-converge shape, unknown agent, enforcement gap without the flag, existing log without `force`.

### chit_converge_next

Run exactly one implement to review iteration, blocking until it settles; emits a heartbeat while it runs. Input: `loop_id`. A normal iteration returns `{ iteration, verdict, decision, findingCount, checksRun, changedFiles, workspaceWarnings, usage?, auditRunId?, stopStatus?, loop }` (`changedFiles` is the task-relevant diff; `workspaceWarnings` flags untracked generated artifacts left in the tree; `usage` is the iteration's token/cost when the adapters reported it). A set `stopStatus` means the loop also stopped this round: `proceed` converges, `block` blocks, exhausting the budget is `max-iterations`. A cancelled iteration returns `{ cancelled: true, iteration, loop }` and records a clean `cancelled` stop with no iteration record, never a fake-successful round. A graceful manifest failure returns `{ failed: true, iteration, failure, loop }` and closes the loop `blocked`. It rejects a loop that already stopped or already has an iteration in flight (the loop log is single-writer).

### chit_converge_status

Compact control-plane view, answering "what should I do next?". Input: `loop_id`. Returns `describeConverge` from the in-memory session alone (no loop-log read), so it is cheap to poll.

### chit_converge_cancel

Cancel a loop. Input: `loop_id`. If an iteration is in flight, aborts it (it settles as a clean `cancelled` stop); if the loop is open but idle, closes it `cancelled` now; a loop that already stopped is reported back unchanged. Returns `{ cancel: { state }, loop }`, where `state` is `cancelling`, `closed`, or `already`.

### chit_converge_trace

Diagnostic history, answering "what happened?". Input: `loop_id`. Reads straight from the durable loop log (not a second source of truth) and adds the live state. Returns `{ loop_id, status, active, audit_refs[], records[] }`, where `records` are the loop-log records: the header, each iteration (summary, changed files, verdict, decision, usage, audit ref), and the stop record.

## Background jobs

The converge tools above are foreground: one blocking iteration per call, so you checkpoint each round. A background **job** is the other mode: a detached worker advances the loop on its own while you keep working, and you inspect or stop it from any later turn. Use foreground converge when you want to review every iteration; use a job when the task is scoped enough to run unattended. A job's state is durable (it survives an MCP reconnect): the job record lives under the state dir, the iterations in the loop log, the transcripts in the audit store.

### chit_converge_run

Start a converge loop as a background job. Inputs: `task`, `scope`, `cwd?`, `manifest_path?`, `max_iterations?` (default 3), `loop_id?`, `force?`, `allow_unenforced_permissions?`. Validates the manifest synchronously, reserves the loop, spawns the detached worker, and returns immediately with `{ jobId, loopId, repo, state: "queued", nextAction, warnings? }`. v1 starts a NEW loop only: an existing `loop_id` is refused (continue a foreground loop with `chit_converge_next`, or pass `force: true` / a new `loop_id`).

### chit_job_status

Inspect one job. Input: `job_id`. Returns the job's `state` (`queued` / `running` / `completed` / `cancelled` / `failed`) plus a derived `stale` (a running job whose worker is gone or silent), the current `phase`, `iterationsCompleted`, `lastVerdict`, `stopStatus`, `auditRefs`, and the latest iteration's `changedFiles` / `workspaceWarnings` / `usage` (read from the loop log). For operating long jobs it also reports timing in milliseconds: `elapsedMs` (run duration), `lastHeartbeatAgeMs` (how long since the worker last beat, while in flight), and `phaseElapsedMs` (how long it has been in the current phase). Read-only.

### chit_job_cancel

Cancel a job from any turn. Input: `job_id`. Persists the cancel intent first (so it survives a worker restart), then signals the worker's process group. A queued job is cancelled before it starts; a running job stops at the next safe point and records a clean `cancelled` stop. A finished job is reported back unchanged. `chit_status` lists active and recent jobs alongside the in-memory runs.

## Batches

A **batch** runs several converge tasks in parallel, one per git worktree, as background jobs. It is a thin coordinator over the background jobs above: it plans a task graph, creates a worktree per task, and launches a `chit_converge_run` job per runnable task. It owns no execution and never auto-merges; the deliverable is a set of reviewable worktree branches. There is no daemon: progress happens only at explicit tool calls. Batch state is durable under the state dir, keyed by repo (not in the reviewed tree).

You hand chit a reviewed task graph; chit runs it. Each task declares `claimedPaths` (the files it will touch); tasks with overlapping claims never run concurrently (they serialize into later waves), so parallel tasks cannot race on the same files. A task may declare `dependencies` (task ids that must reach `review_ready` first). The converge manifest per task resolves as task `manifestPath` > batch `manifest_path` > the bundled default, so a batch can mix model pairs (a Codex-writer task beside a Claude-writer task) by pointing each at the right manifest.

### chit_batch_start

Plan the graph, create worktrees + launch the initial runnable wave, return immediately. Inputs: `tasks` (each `{id, title, body, dependencies?, claimedPaths?, allowPathOverlap?, manifestPath?}`), `cwd?`, `max_parallel?` (default 2), `base_branch?` (default HEAD), `manifest_path?`, `max_iterations?`. `claimedPaths` is required per task unless `allowPathOverlap` is set (which makes the task run alone). Returns the batch overview (see status).

### chit_batch_list

List the batches in this repo, newest first: `id`, `status`, task count, how many tasks are `review_ready` / `failed`, `createdAt`, and `cleanedAt` if it has been cleaned up. Use it to recover a batch id you lost, then `chit_batch_status` for the full view. Read-only. Inputs: `limit?` (newest N), `cwd?`.

### chit_batch_status

Read-only overview: each task's status, live job state/phase, branch/worktree path, changed files, audit refs, plus `runnableCount` and a `nextAction`. **Inspection is safe**: this never launches jobs, creates worktrees, or mutates state. Inputs: `batch_id`, `cwd?`.

### chit_batch_advance

The progression trigger. Reconciles finished jobs into task state (`converged` -> `review_ready`; `blocked` / `max-iterations` / failed / stale -> `failed`; a dependent proceeds only past a `review_ready` task), then launches the next runnable wave. Call it when status reports a finished job or runnable tasks. Inputs: `batch_id`, `cwd?`.

### chit_batch_cancel

Request cancellation of every active task job (intent-first, same safety as `chit_job_cancel`) and mark pending tasks cancelled. Running jobs settle cleanly in the background; worktrees are left in place for inspection. Inputs: `batch_id`, `cwd?`.

### chit_batch_cleanup

Retire a batch's worktrees and branches once you are done reviewing them. **Safe by default**: with `confirm` omitted/false it is a DRY RUN that lists which worktrees/branches would be removed and which changed-file diffs that would discard, and removes nothing. With `confirm: true` it removes them (`git worktree remove --force` + `branch -D`). Refuses while any task is still running. **Never deletes the batch/job/loop/audit receipts** — those stay as durable history (it records `cleanedAt` on the batch). Inputs: `batch_id`, `confirm?` (default false), `cwd?`.

## Audit tools

The audit tools read the local audit transcripts that audited runs write (`chit converge`, `chit run --audit`, MCP `chit_run_start audit:true`, and MCP converge), so a run's prompts, outputs, usage, and timeline are inspectable from the chat. Same reader as the CLI `chit audit list/show`. Read-only: a run with no `run.completed` event is reported `incomplete` with the reason from the timeline alone (an open call killed mid-flight, a failed step, or an abandoned run). Bodies are read only through blob refs a run's own events carry, never a caller-supplied path, so inspection can never serve an arbitrary file.

### chit_audit_list

List audited runs, newest first. Input: `limit?`. Returns `{ runs[] }`, each run `{ runId, manifestId, surface, scope?, loopId?, iteration?, startedAt?, status, stepCount, usage?, openCall? }`, where `status` is the `run.completed` status or `incomplete`, and `openCall` (when present) names a step whose adapter call started but never completed (killed mid-call).

### chit_audit_show

Show one run. Inputs: `run_id`, `include_bodies` (default `false`). Returns `{ summary, incompleteReason?, participants?, timeline[] }`: the summary above, the reason when incomplete, the participant config recorded at start, and the structured event timeline. Prompt/output/event bodies are attached to their timeline entries (`input`/`output`/`raw`) only when `include_bodies` is true, since they can be large or hold secrets. Errors on an invalid or missing run id.

## Observability (heartbeat)

While a call step runs, `chit_run_step` emits, every ~5s, both a progress notification (with `progressToken`) and a logging notification carrying the same latest-state text: `"<step> · <participant> (<agent>) still running · Ns elapsed"`. Claude Code renders the latest heartbeat live in the collapsed tool call; the full transcript is `chit_run_trace`.

There is no within-step streaming of the agent's output to the MCP client. The heartbeat is latest-state text, not a token stream, and `chit_run_step` returns only the step's final output. On an audited run the adapter does capture the agent's live event stream as `adapter.event` records, but that feeds the [audit log](/docs/audit-log), not the MCP client.

## Cancellation

Cancellation has two reachable paths. The portable one is `chit_run_cancel`: `chit_run_step` owns an `AbortController` per run and step, registered for the whole call; `chit_run_cancel` aborts it, both adapters (`claude-cli`, `codex-exec`) kill their child process on abort and reject, and the engine discriminates on `signal.aborted` to settle the step `cancelled`. It is proven end-to-end against codex, where a concurrent `chit_run_cancel` killed an in-flight step in ~6s rather than the ~220s it would otherwise have taken. The second path is ambient: in Claude Code, pressing **Esc** during a blocking `chit_run_step` propagates request cancellation through the call's folded-in `extra.signal`, which aborts the same controller. A live probe confirmed this (a long codex step, Esc after the first heartbeat, settled `cancelled` "aborted by client" in ~5.4s, with no `chit_run_cancel` call). Esc behavior is client-specific, so `chit_run_cancel` stays the portable backup.

`chit_converge_next` follows the same contract: it folds the request's `extra.signal` into the in-flight iteration's abort, so Esc (or `chit_converge_cancel`) cancels the running implement or review. A cancelled iteration records a clean `cancelled` stop with no iteration record, so the loop log never carries a fake-successful round.

## Limits

- Runs live in an in-memory store (`run-store.ts`). A server restart or reconnect loses them. The store is idle-evicting: a run untouched for more than 1h is dropped on the next `chit_run_start` sweep, unless it still has a `running` step (those are never evicted). Cleanup is opportunistic, so memory is bounded by future starts, not by wall-clock alone.
- Converge sessions live in the same kind of in-memory store, idle-evicted on the next `chit_converge_start` sweep (a loop with an in-flight iteration is never evicted). The durable loop log persists regardless; only the in-memory session is dropped, after which `chit_converge_status` and `chit_converge_trace` no longer find that `loop_id`.
- `inputs` are string to string. `file[]` inputs are not expressible via MCP.
- Concurrent `per_scope` steps would hit the session store's read-modify-write race, so keep same-scope steps serial.

### Not supported yet

Client-facing output streaming (a live token stream to the MCP client). The heartbeat is enough for now. Live adapter event capture for the audit log is separate, and has shipped.
