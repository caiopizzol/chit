Chit turns the agent workflow you run by hand into config. You declare a routine; Chit resolves the models, runs each step, passes context forward, checks the result, and writes a receipt. Roles like implementer, reviewer, and planner are not built in. They are names and prompts you define.

This page takes you from zero to your first run. For every field, see the [Config reference](/docs/config).

## Before you start

Chit shells out to the agent CLIs you already use, so install and sign in to at least one:

- `claude`
- `codex`
- `gemini`

Chit installs and runs under [Bun](https://bun.sh). Step 3 (`chit doctor`) checks all of this for you.

## 1. Install Chit

```sh
bun add -g @chit-run/cli
```

This puts the `chit` command on your PATH. Move into the project you want to work in:

```sh
cd /path/to/your-project
```

## 2. Create your first routine

```sh
chit init implement --template loop
```

This writes `chit.config.json` with:

- **starter profiles** `claude` and `codex`, the local adapter bindings a routine calls.
- an **`implement` routine**: a `builder` (Claude, read-write) makes the change, a `reviewer` (Codex, read-only) returns a structured `{ passed, issues }` verdict, and a `verify` check runs your command. It loops until the check passes and the review passes.

Edit two things before you run:

- the **check** is a placeholder (`true # replace ...`). Swap it for your real command, such as `bun test`.
- the **profiles** use each adapter's default model. Point them at the models you want, for example `"builder": { "adapter": "claude", "model": "claude-opus-4-8", "effort": "max" }`.

The [Config reference](/docs/config) documents every field.

## 3. Check readiness

```sh
chit doctor
```

Confirms the config parses, the routines resolve, and your agent CLIs are on PATH. Add `--real` to probe that each adapter can actually reach its model.

## 4. Inspect before spending model calls

```sh
chit inspect implement
```

Resolves the routine, binds each agent to a real adapter and model, and prints the steps and limits. An invalid config fails here, before any model runs.

## 5. Run it

```sh
chit run implement --input task="add a --version flag"
```

A routine with a check or a read-write agent runs in a disposable git worktree. It is a dry run: it produces a patch and stops. Your working tree is untouched.

## 6. Review, then apply

```sh
chit trace --full <run-id>
chit apply <run-id>
```

`trace` shows what each step and iteration did, with the resolved bindings and the saved patch. `apply` replays that exact patch through a gate: same HEAD, clean tree, `git apply --check`.

## Which command do I run?

| Goal | Command |
|---|---|
| Check setup | `chit doctor` |
| List routines | `chit routines` |
| Understand a routine | `chit inspect implement` |
| Run safely (dry run) | `chit run implement --input task="..."` |
| List past runs | `chit runs` |
| Review a result | `chit trace --full <run-id>` |
| Apply a patch | `chit apply <run-id>` |

## How a run is derived

You do not choose an execution mode. Chit reads the routine shape.

| If the routine has | Chit runs |
|---|---|
| read-only calls and format steps | a text run in the current directory |
| `routine` steps | a flow that passes outputs forward |
| `repeat` | a loop until the stop condition is met |
| checks or read-write agents | a sandboxed run in a disposable git worktree |

A sandboxed run is dry-run by default: a patch you inspect, then `chit apply` writes the exact reviewed result.

## Patterns to copy

These are examples, not built-in modes.

| Pattern | Example | What it shows |
|---|---|---|
| Grill before planning | `examples/feature-griller.json` | Similar shape to [Matt Pocock `grill-me`](https://github.com/mattpocock/skills/blob/main/skills/productivity/grill-me/SKILL.md): question the plan before implementation. |
| Goal loop | `examples/goal.json` | Similar shape to [Claude `/goal`](https://code.claude.com/docs/en/goal) and [Codex Goals](https://developers.openai.com/cookbook/examples/codex/using_goals_in_codex): draft, judge, continue until ready. |
| Multi-model panel | `examples/panel-review.json` | Similar shape to [OpenRouter Fusion](https://openrouter.ai/docs/guides/features/server-tools/fusion): two models answer independently; a judge compares them. |
| Build and review loop | `examples/implementation-review.json` | One model writes in a sandbox, another reviews, and checks gate convergence. |
