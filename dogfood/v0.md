# Dogfood v0

The durable record of running real chits against real agents. Feature work on
Studio is paused until this milestone passes (see the Slice 4 milestone note).

Unlike Pickled, chit runs spend tokens and are non-deterministic, so the real
matrix is **manual, not CI**. CI keeps the fast fake-CLI tests (408 passing).
This file holds the scenarios, exact commands, pass/fail criteria, and known
blockers. Each run drops a receipt in `receipts/` (template at the bottom).

## The truth test

> Claude proposes, Codex verifies, Claude continues.

If that flow still feels like copy-paste between two terminals, chit is not
ready. Everything below builds up to being able to judge that.

## Exact commands

```sh
# 1. consult-stateless (CLI). Parallel fan-out to codex + claude, format merge.
#    --allow-unenforced-permissions is REQUIRED: claude declares read_only,
#    which claude-cli cannot enforce, so the run exits 2 without it (see 0001).
bun apps/cli/src/cli/run.ts run examples/consult-stateless.json \
  --allow-unenforced-permissions \
  --input question="In one sentence, what is a monorepo?"

# 2/3. single agent. ask-claude needs --allow-unenforced-permissions (read_only);
#      ask-codex does not (codex-exec enforces read_only via --sandbox read-only).
bun apps/cli/src/cli/run.ts run examples/ask-codex.json  --input question="..."
bun apps/cli/src/cli/run.ts run examples/ask-claude.json --allow-unenforced-permissions --input question="..."

# 4. consult with per_scope sessions. CLI requires --scope to satisfy
#    can_provide_stable_scope; run twice with the same scope to test resume.
bun apps/cli/src/cli/run.ts run examples/consult.json \
  --scope dogfood-1 --input question="..."

# 5. THE COPY-PASTE A/B. Install the propose -> verify -> revise chit as a
#    Claude Code skill, then invoke it from INSIDE a Claude Code session. It
#    must go in the REAL skills dir so Claude Code can see it (a temp dir or
#    CHIT_SKILLS_DIR will not work: the CLI ignores CHIT_SKILLS_DIR, and Claude
#    Code only reads ~/.claude/skills). Uninstall when done.
bun apps/cli/src/cli/run.ts install \
  dogfood/propose-verify-revise.json --as claude-skill --allow-unenforced-permissions
#    then, inside Claude Code: /propose-verify-revise <your task>
#    (disable-model-invocation: true, so you invoke it explicitly).
bun apps/cli/src/cli/run.ts uninstall propose-verify-revise   # cleanup

# To verify install MECHANICS in isolation (not the in-session invoke), use
# --to a temp dir; this keeps the real skills dir clean:
#   bun apps/cli/src/cli/run.ts install dogfood/propose-verify-revise.json \
#     --as claude-skill --allow-unenforced-permissions --to "$(mktemp -d)"

# 6-8. Studio loop (manual). Launch, edit, Save, Install via the header modal,
#      then invoke in Claude Code. For conflict (7): edit the file on disk after
#      loading, then Save/Install and confirm the 409 banner.
bun apps/cli/src/cli/run.ts studio examples/consult.json
```

## Transcript (`--trace`)

Add `--trace` to any `chit run` to render a step transcript to stderr (per step:
participant, agent, session policy, elapsed, status, prompt/output previews). It
streams live in a terminal; via a skill it is buffered and shown ahead of the
final output. `chit install --trace` bakes it into the generated skill so the
transcript shows in chat. Used to test whether the skill "black box" complaint
is fixed by transparency; see receipt 0003 and the conversational-handoff
proposal. Full payloads stay in `result.trace`; the renderer shows previews.

## Pass / fail criteria

- **Run completes**: exit 0, the `format` step's output is printed, and it
  contains both agents' answers under their headings.
- **Parallel**: `consult-stateless` dispatches codex and claude concurrently
  (both calls overlap, not strictly serialized).
- **Session resume (4)**: a second run with the same `--scope` continues the
  prior conversation rather than starting cold.
- **Skill loop (5)**: the skill is invokable in Claude Code by name, takes the
  one string argument, and runs the chit without manual copy-paste.
- **Conflict (7)**: editing the file underneath a loaded Studio doc makes
  Save/Install return 409 and the UI shows the conflict banner.
- **Permission warning (8)**: installing a chit with a claude `read_only`
  participant shows the unenforced-permission consent and, once installed, the
  generated skill warns on every run.

## Scenario 5: the copy-paste A/B (the truth test)

The honest way to test "does chit kill the copy-paste" is to do the task BOTH
ways and compare. Same task, back to back.

**A. Baseline (what you do now).** Inside Claude Code, do the routine by hand:
ask Claude to propose, copy its answer, switch to a Codex terminal, paste and
ask it to find problems, copy the critique, switch back to Claude, paste it and
ask Claude to revise. Note: number of copy-pastes, number of window switches,
elapsed time, and any point where you had to re-explain context.

**B. chit (one invocation).** Install once (real skills dir, see commands), then
inside the same Claude Code session:

```
/propose-verify-revise <the same task, in one line>
```

It runs claude (propose) -> codex (verify) -> claude (revise, continuing its own
session) and returns one markdown block with all three parts. No switching, no
pasting. Uninstall when done.

**What to judge (write into receipt 0003):**

- Copy-pastes and window switches: baseline count vs chit (chit should be ~zero).
- Did "claude continues" actually work? The `revise` step does NOT re-paste the
  proposal; it relies on `per_scope` session resume. If Claude's revision
  ignores what it proposed, session continuity failed. (Verified to work in
  code and in receipt 0002 across runs; this confirms it within one run.)
- Did codex actually get the proposal? Its critique should reference specifics
  from Claude's proposal (that is the explicit handoff the template does).
- One-shot-per-step vs interactive: the manual flow lets you nudge each agent
  mid-thought; chit gives each step one prompt. Did that rigidity hurt, or was
  the single pass enough? This is the "manifest shape too narrow?" question and
  the fairest thing to watch.
- Did the merged output read as well as what you would have assembled by hand?

If B still feels like copy-paste, or the one-shot steps make the answer worse
than the manual back-and-forth, chit is not ready and the finding goes in the
receipt.

## Known blockers (verified, not assumed)

- **`file[]` inputs do not run on the CLI.** `chit run` refuses any manifest
  that declares a `file[]` input (`can_pass_files` is not provided by the CLI
  surface). This is why `investigate-bug.json` is **not** a fair first smoke;
  it also uses `per_scope`, so it would additionally need `--scope`.
- **Claude Code skill takes exactly one string input.** The surface throws if a
  manifest declares more than one input, or if the primary input is not
  `string` (`apps/cli/src/surfaces/claude-skill.ts`). `consult` /
  `consult-stateless` (one `question` input) install fine; `investigate-bug`
  (two inputs incl. `file[]`) does not.
- **Claude cannot enforce `filesystem: read_only`.** Any chit with a claude
  participant hits the permission-warning path. This gates **`chit run` too**,
  not just install: a plain `chit run consult-stateless.json` exits 2 before
  reaching the agents. It needs `--allow-unenforced-permissions` (CLI) or the
  consent checkbox (Studio). First-run friction for anyone using a claude chit.
- **Real runs are manual.** They spend tokens and are non-deterministic, so
  they live here, not in CI.
- **`CHIT_SKILLS_DIR` only isolates Studio, not the CLI.** The CLI
  `install` / `list` / `uninstall` honor `--to <dir>` (default
  `~/.claude/skills`) and ignore `CHIT_SKILLS_DIR`; only Studio's lifecycle
  reads the env var (`run.ts` `buildStudioLifecycle`). So `CHIT_SKILLS_DIR=...
  chit install` silently writes to the REAL skills dir. Use `--to` to isolate a
  CLI install. This bit the first draft of scenario 5; recorded as a finding.
